# Telegram Moderation Bot

Модульный Telegram-бот для модерации групп и супергрупп: антиспам (ссылки,
запрещённые слова, повторы), глобальный список скамеров, правило "пригласи
друга", гибкая настройка наказаний и языка интерфейса — всё per-chat, через
команды в самом чате.

Весь код бота находится в одном файле — `bot.py`. Внутри он разбит на
пронумерованные секции (см. оглавление в начале файла): конфигурация,
модели БД, репозитории, ядро pipeline, утилиты, i18n, модули модерации,
наказания, admin-команды, точка входа.

## Возможности

- **Антиспам**: три независимо включаемых правила
  - **Ссылки** — белый список доменов/каналов и белый список отправителей;
    распознаёт обфускацию (`example[.]com`, `example dot com`, гомоглифы).
  - **Запрещённые слова** — список фраз per-chat, без учёта регистра,
    с защитой от обхода через повтор букв (`сууукааа` → `сука`).
  - **Повторы** — детект дублирующихся сообщений в настраиваемом окне
    времени, per-topic или cross-topic (между темами супергруппы).
- **Глобальный список скамеров**: один список на всего бота, управляется
  глобальными админами, применяется во всех чатах сразу.
- **"Пригласи друга"**: новый участник обязан пригласить кого-то за
  отведённое время, иначе получает наказание; правило сбрасывается при
  повторном входе после выхода.
- **Гибкие наказания**: удаление сообщения, бан, мут (24ч/навсегда), кик —
  настраивается отдельно для каждого правила через `/set_punishment`.
- **Мультиязычность**: русский и английский из коробки; новый язык — это
  просто новый файл `locales/xx.yml`, без изменений кода.

## Установка и запуск

```bash
cp .env.example .env
# отредактируйте .env: BOT_TOKEN, DATABASE_URL, GLOBAL_ADMIN_IDS

pip install -r requirements.txt
alembic upgrade head
python bot.py
```

Требуется PostgreSQL (для тестов используется in-memory SQLite).

## Тесты

```bash
pip install -r requirements-dev.txt
pytest tests/ -v
```

134 unit-теста покрывают всю бизнес-логику: чекеры антиспама, наказания,
права доступа, i18n, admin-команды, фоновые задачи, graceful shutdown.

## Структура репозитория

```
bot.py              # весь код бота (см. оглавление секций в начале файла)
alembic/            # миграции БД
locales/            # ru.yml, en.yml
tests/              # 134 теста
.env.example        # список переменных окружения с комментариями
requirements.txt          # зависимости для запуска
requirements-dev.txt        # + зависимости для тестов
```

## Добавление нового модуля модерации

1. В `bot.py` реализовать класс с интерфейсом `ModerationModule` (секция 5):
   `name`, `handled_events`, `is_enabled()`, `check()` — по образцу
   `ScamlistModule` (секция 9, самый компактный пример).
2. Зарегистрировать в `_register_modules()` (секция 12) одной строкой.
3. Добавить admin-команды в секцию 11, если модулю нужна настройка из чата.
4. Добавить ключи переводов в `locales/*.yml`.

## Добавление нового языка

Создать `locales/xx.yml` по образцу `locales/ru.yml` — код менять не нужно,
`TranslationLoader` сканирует директорию при старте.

## Известные ограничения и зафиксированные решения

- **Redis не используется** — история сообщений для антидубликатов и весь
  остальной стейт хранятся в Postgres; TTL реализован периодической job
  (`HISTORY_CLEANUP_INTERVAL_SECONDS`), а не через встроенный TTL Redis.
- **Добавление скамера только по username (без известного user_id)**
  не поддерживается: Telegram Bot API не даёт надёжного способа получить
  user_id по username без предварительного взаимодействия с пользователем.
  `/scamlist_add` принимает user_id, ссылку на профиль (`t.me/username`)
  или `@username` — в последнем случае поиск идёт по уже известным записям.
- **Права проверяются "живьём" через Bot API**, а не только по кешу в БД —
  на случай, если админа разжаловали, а локальный кеш ещё не обновился.
- **Наказание "kick"** реализовано как `ban_chat_member` + немедленный
  `unban_chat_member(only_if_banned=True)`.
- **join_required** проверяется двумя путями: как safety-net при следующем
  сообщении "просроченного" пользователя, и как отдельная периодическая
  фоновая задача для "тихих" нарушителей, которые вообще ничего не пишут.
- Репозитории намеренно НЕ используют raw `INSERT ... ON CONFLICT DO UPDATE`
  для апдейтов существующих записей — вместо этого паттерн get-or-create +
  мутация ORM-объекта (причина задокументирована в коде рядом с
  `JoinRequiredStateRepository.set_pending`).
